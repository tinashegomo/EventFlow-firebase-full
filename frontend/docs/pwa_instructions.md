## 1. What Is a PWA — Quick Overview

A Progressive Web App adds three capabilities on top of your existing web app:

| Capability | What It Means for EventFlow |
|---|---|
| **Installable** | Users can add EventFlow to their home screen on Android, iOS, and desktop — it opens like a native app with no browser chrome |
| **Offline-capable** | Core pages and assets load even without internet. Firestore's local cache keeps event data available |
| **Push Notifications** | You can send event reminders and alerts to users via Firebase Cloud Messaging, even when the app is closed |

A PWA is **not** a separate build or a new codebase. It is the same Vite app, enhanced with a Web App Manifest and a Service Worker.

---

## 2. What Changes and What Stays the Same

### ✅ Stays exactly the same
- All existing component code
- All existing routes
- All Firebase SDK calls (Auth, Firestore, Storage)
- All Tailwind classes, CSS variables, theme colors, fonts, and design tokens
- All existing page layouts and UI components

### 🔧 What gets added
- `vite-plugin-pwa` package (dev dependency only)
- PWA config block inside `vite.config.js`
- `public/manifest.webmanifest` (auto-generated by the plugin)
- App icon files in `public/` (192px and 512px PNG)
- A service worker (auto-generated by the plugin using Workbox)
- `public/firebase-messaging-sw.js` (only if you use push notifications)
- Optional: an "Add to Home Screen" prompt component

## 3. Prerequisites

Before starting, make sure you have:

- Node.js 18+ and npm/yarn
- Vite 4+ (already in EventFlow)
- Firebase project already configured
- HTTPS in production (required for PWAs — Firebase Hosting provides this automatically)
- A square logo or icon image (at least 512×512px) to generate app icons from

## 4. Phase 1 — Install & Configure vite-plugin-pwa

### Step 1.1 — Install the plugin

```bash
npm install -D vite-plugin-pwa
```

### Step 1.2 — Update `vite.config.js`

Open your existing `vite.config.js`. Import and add the PWA plugin. **Do not change anything else in this file.**

```js
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import { VitePWA } from 'vite-plugin-pwa'

export default defineConfig({
  plugins: [
    react(),
    VitePWA({
      registerType: 'autoUpdate',

      // Assets to pre-cache beyond the JS/CSS bundle
      includeAssets: [
        'favicon.ico',
        'apple-touch-icon.png',
        'masked-icon.svg'
      ],

      // Web App Manifest
      manifest: {
        name: 'EventFlow',
        short_name: 'EventFlow',
        description: 'Professional event management platform',
        theme_color: '#YOUR_BRAND_COLOR',       // ← replace with your existing primary color
        background_color: '#YOUR_BG_COLOR',     // ← replace with your existing background color
        display: 'standalone',
        orientation: 'portrait-primary',
        scope: '/',
        start_url: '/',
        icons: [
          {
            src: 'pwa-192x192.png',
            sizes: '192x192',
            type: 'image/png'
          },
          {
            src: 'pwa-512x512.png',
            sizes: '512x512',
            type: 'image/png'
          },
          {
            src: 'pwa-512x512.png',
            sizes: '512x512',
            type: 'image/png',
            purpose: 'any maskable'
          }
        ]
      },

      // Workbox service worker config
      workbox: {
        // Cache all built assets
        globPatterns: ['**/*.{js,css,html,ico,png,svg,woff,woff2}'],

        // Do NOT cache Firebase Auth iframe or API calls
        navigateFallbackDenylist: [
          /^\/__\/auth/,       // Firebase Auth hosted UI
          /^\/api\//,          // Any REST API routes
        ],

        // Runtime caching rules
        runtimeCaching: [
          {
            // Cache Google Fonts
            urlPattern: /^https:\/\/fonts\.googleapis\.com\/.*/i,
            handler: 'CacheFirst',
            options: {
              cacheName: 'google-fonts-cache',
              expiration: { maxEntries: 10, maxAgeSeconds: 60 * 60 * 24 * 365 }
            }
          },
          {
            // Cache Firebase Storage images
            urlPattern: /^https:\/\/firebasestorage\.googleapis\.com\/.*/i,
            handler: 'StaleWhileRevalidate',
            options: {
              cacheName: 'firebase-storage-cache',
              expiration: { maxEntries: 50, maxAgeSeconds: 60 * 60 * 24 * 7 }
            }
          }
        ],

        // Don't let Workbox touch the FCM service worker
        exclude: ['firebase-messaging-sw.js']
      },

      // Dev options — enables SW in development for easier testing
      devOptions: {
        enabled: false  // Set to true temporarily when testing SW locally
      }
    })
  ]
})
```

> **Note on `theme_color`:** This controls the browser UI bar color on Android when the app is installed. Use your existing primary brand color hex code (e.g. `#1E40AF`). It should not be white unless that matches your brand.

## 5. Phase 2 — App Icons & Manifest

### Step 2.1 — Generate Icons

You need these files in your `public/` folder:

| File | Size | Used For |
|---|---|---|
| `pwa-192x192.png` | 192×192px | Android home screen, splash screen |
| `pwa-512x512.png` | 512×512px | High-res Android, desktop install |
| `apple-touch-icon.png` | 180×180px | iOS home screen (Safari) |
| `favicon.ico` | 32×32px | Browser tab (you likely have this already) |
| `masked-icon.svg` | Any SVG | Monochrome icon for some Android launchers |

**Option A — Use pwa-asset-generator (recommended)**

```bash
npx pwa-asset-generator ./src/assets/your-logo.png ./public \
  --manifest ./public/manifest.webmanifest \
  --index ./index.html \
  --background "#YOUR_BG_COLOR" \
  --padding "10%"
```

This generates all required sizes from a single source image.

**Option B — Manual via realfavicongenerator.net**

1. Go to https://realfavicongenerator.net
2. Upload your 512×512 logo
3. Download the package
4. Place the PNG files into `public/`

### Step 2.2 — Link Icons in `index.html`

Add these inside the `<head>` of your `index.html` if not already present:


## 6. Phase 3 — Service Worker & Caching Strategy

The plugin auto-generates a service worker using **Workbox**. You do not write it by hand. Here is how the caching layers work:

### Pre-cache (at install time)
All files matching `globPatterns` in the built bundle are pre-cached. This means your JS chunks, CSS, fonts, and icons are stored immediately when the user first visits — so subsequent loads (and offline loads) are instant.

### Runtime Cache — StaleWhileRevalidate
Firebase Storage assets and images are served from cache immediately while being refreshed in the background. This gives instant image loads while staying up to date.

### Not cached
- Firebase Auth requests (`/__/auth/`)
- Firestore reads/writes (these go through Firebase's own offline layer — see Phase 5)
- Any POST/PUT/DELETE requests (mutations are never cached)

### Update behavior (`registerType: 'autoUpdate'`)
When you deploy a new build, the service worker detects the change and updates automatically on the next page load. No user action required.

### Optional — Show an update notification

If you want to inform users when an update is ready, add this to your `main.jsx`:

```jsx
import { registerSW } from 'virtual:pwa-register'

registerSW({
  onNeedRefresh() {
    // You can show a toast or banner here using your existing UI components
    console.log('New version available — reloading...')
  },
  onOfflineReady() {
    console.log('EventFlow is ready for offline use')
  }
})
```

## 7. Phase 4 — Firebase-Specific Setup

Firebase introduces two service worker conflicts to handle carefully.

### 4.1 — Firebase Auth & the Service Worker

Firebase Auth uses a hidden iframe hosted at `/__/auth/handler`. If the service worker intercepts this, sign-in breaks. The `navigateFallbackDenylist` config in Phase 1 already excludes this path — no extra work needed.

### 4.2 — Firebase Messaging Service Worker

If you use Firebase Cloud Messaging (push notifications), Firebase requires its own dedicated service worker file at the root: `public/firebase-messaging-sw.js`. This **cannot** be inside Workbox's generated SW — it must be a separate file.

Create `public/firebase-messaging-sw.js`:

```js
// Use the compat versions — they work as importScripts in service workers
importScripts('https://www.gstatic.com/firebasejs/10.12.0/firebase-app-compat.js')
importScripts('https://www.gstatic.com/firebasejs/10.12.0/firebase-messaging-compat.js')

firebase.initializeApp({
  apiKey: 'YOUR_API_KEY',
  authDomain: 'YOUR_PROJECT.firebaseapp.com',
  projectId: 'YOUR_PROJECT_ID',
  storageBucket: 'YOUR_PROJECT.appspot.com',
  messagingSenderId: 'YOUR_SENDER_ID',
  appId: 'YOUR_APP_ID'
})

const messaging = firebase.messaging()

// Handle background messages (app is closed or in background)
messaging.onBackgroundMessage((payload) => {
  const { title, body, icon } = payload.notification

  self.registration.showNotification(title, {
    body,
    icon: icon || '/pwa-192x192.png',
    badge: '/pwa-192x192.png',
    data: payload.data
  })
})

// Handle notification click — open the app to the right page
self.addEventListener('notificationclick', (event) => {
  event.notification.close()
  const targetUrl = event.notification.data?.url || '/'
  event.waitUntil(clients.openWindow(targetUrl))
})
```

> ⚠️ **Important:** The `exclude: ['firebase-messaging-sw.js']` line in the vite config (Phase 1) prevents Workbox from overwriting this file during build.

## 8. Phase 5 — Offline Support with Firestore

Firestore has **built-in offline persistence** — it caches documents locally in IndexedDB and queues writes for sync when back online. Enable it once in your Firebase initialization file.

### Step 5.1 — Enable Persistent Cache

Find your Firebase init file (e.g. `src/firebase.js` or `src/lib/firebase.js`) and update the Firestore initialization:

```js
import { initializeApp } from 'firebase/app'
import {
  initializeFirestore,
  persistentLocalCache,
  persistentMultipleTabManager
} from 'firebase/firestore'

const app = initializeApp(firebaseConfig)

// Replace getFirestore(app) with this:
export const db = initializeFirestore(app, {
  localCache: persistentLocalCache({
    tabManager: persistentMultipleTabManager() // Supports multiple browser tabs
  })
})
```

This replaces a plain `getFirestore(app)` call. No other changes needed in your components.

### What this gives you

| Scenario | Behaviour |
|---|---|
| User goes offline mid-session | Firestore reads still return cached data; writes are queued |
| User creates/updates an event offline | Change is saved locally and synced automatically on reconnect |
| User closes the app and reopens offline | Last fetched data is available immediately |
| Multiple browser tabs open | All tabs share a single cache safely |

### Step 5.2 — Show an Offline Indicator (Optional)

You can add a simple banner using your existing design components — no new styles needed:

```jsx
// src/hooks/useOnlineStatus.js
import { useState, useEffect } from 'react'

export function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(navigator.onLine)

  useEffect(() => {
    const handleOnline = () => setIsOnline(true)
    const handleOffline = () => setIsOnline(false)
    window.addEventListener('online', handleOnline)
    window.addEventListener('offline', handleOffline)
    return () => {
      window.removeEventListener('online', handleOnline)
      window.removeEventListener('offline', handleOffline)
    }
  }, [])

  return isOnline
}
```

Then use it anywhere in your layout:

```jsx
import { useOnlineStatus } from '../hooks/useOnlineStatus'

function OfflineBanner() {
  const isOnline = useOnlineStatus()
  if (isOnline) return null
  return (
    <div className="/* use your existing warning/alert styles */">
      You are offline. Changes will sync when reconnected.
    </div>
  )
}
```

## 9. Phase 6 — Install Prompt (Add to Home Screen)

Browsers show an install prompt automatically, but you can create a custom one that fits your existing UI.

### Step 6.1 — Create the hook

```jsx
// src/hooks/useInstallPrompt.js
import { useState, useEffect } from 'react'

export function useInstallPrompt() {
  const [installPrompt, setInstallPrompt] = useState(null)
  const [isInstalled, setIsInstalled] = useState(false)

  useEffect(() => {
    // Check if already installed
    if (window.matchMedia('(display-mode: standalone)').matches) {
      setIsInstalled(true)
      return
    }

    const handler = (e) => {
      e.preventDefault()
      setInstallPrompt(e)
    }

    window.addEventListener('beforeinstallprompt', handler)
    window.addEventListener('appinstalled', () => setIsInstalled(true))

    return () => window.removeEventListener('beforeinstallprompt', handler)
  }, [])

  const triggerInstall = async () => {
    if (!installPrompt) return
    await installPrompt.prompt()
    const result = await installPrompt.userChoice
    if (result.outcome === 'accepted') {
      setInstallPrompt(null)
    }
  }

  return { canInstall: !!installPrompt, isInstalled, triggerInstall }
}
```

### Step 6.2 — Add an Install Button

Place this wherever makes sense in your existing UI — a settings page, a sidebar footer, or a dismissible banner. Use your existing button and card styles:

```jsx
import { useInstallPrompt } from '../hooks/useInstallPrompt'

function InstallButton() {
  const { canInstall, isInstalled, triggerInstall } = useInstallPrompt()

  if (isInstalled || !canInstall) return null

  return (
    <button
      onClick={triggerInstall}
      className="/* your existing primary button classes */"
    >
      Install EventFlow App
    </button>
  )
}
```

> **iOS note:** Safari on iPhone does not fire `beforeinstallprompt`. For iOS users, show a static tooltip saying _"Tap the Share button → Add to Home Screen"_ when `navigator.userAgent` includes `iPhone` or `iPad`.

## 10. Phase 7 — Push Notifications (Optional)

Only implement this if EventFlow needs to send event reminders or alerts.

### Step 7.1 — Request Permission

```js
import { getMessaging, getToken, onMessage } from 'firebase/messaging'
import { app } from './firebase'

const messaging = getMessaging(app)

export async function requestNotificationPermission() {
  const permission = await Notification.requestPermission()
  if (permission !== 'granted') return null

  const token = await getToken(messaging, {
    vapidKey: 'YOUR_VAPID_KEY_FROM_FIREBASE_CONSOLE'
  })

  // Save this token to Firestore under the user's document
  // so your backend can send targeted notifications
  return token
}

// Handle foreground messages (app is open)
export function onForegroundMessage(callback) {
  return onMessage(messaging, callback)
}
```

### Step 7.2 — Get Your VAPID Key

1. Go to Firebase Console → Project Settings → Cloud Messaging
2. Under "Web Push certificates", generate a key pair
3. Copy the public key and paste it as `vapidKey` above

### Step 7.3 — Sending Notifications

Use Firebase Cloud Functions or your backend to send messages via the FCM HTTP API. Each user's saved token is the target. This is outside the scope of the PWA setup itself.

## 11. Phase 8 — Testing

### Local Testing

```bash
# Build the app first — service workers don't run in dev mode
npm run build

# Preview the production build locally
npm run preview
```

Then open `http://localhost:4173` in Chrome.

### Chrome DevTools Checklist

Open DevTools → **Application** tab:

- **Manifest:** Should show your app name, icons, colors, and display mode with no errors
- **Service Workers:** Status should show "activated and running"
- **Cache Storage:** Should list pre-cached assets after first load
- **IndexedDB:** Firestore offline data appears here

### Lighthouse PWA Audit

1. Open DevTools → **Lighthouse** tab
2. Select "Progressive Web App" category
3. Run audit
4. Fix any flagged issues (usually missing meta tags or icon sizes)

### Offline Test

1. In DevTools → Network tab → set throttling to **Offline**
2. Reload the page — core UI should load from cache
3. Navigate between pages — they should work
4. Firestore data should still be readable

### Install Test (Android)

Deploy to Firebase Hosting (HTTPS required) and open in Chrome on Android. You should see "Add to Home Screen" in the browser menu or an automatic install banner.

## 12. Phase 9 — Deployment

### Firebase Hosting (Recommended — already HTTPS)

```bash
npm run build
firebase deploy --only hosting
```

### Add a `firebase.json` Cache Header

Tell Firebase Hosting to serve the service worker without caching so updates propagate immediately:

```json
{
  "hosting": {
    "public": "dist",
    "rewrites": [
      { "source": "**", "destination": "/index.html" }
    ],
    "headers": [
      {
        "source": "/sw.js",
        "headers": [
          { "key": "Cache-Control", "value": "no-cache" }
        ]
      },
      {
        "source": "/firebase-messaging-sw.js",
        "headers": [
          { "key": "Cache-Control", "value": "no-cache" }
        ]
```

## 14. Common Pitfalls & Fixes

| Problem | Cause | Fix |
|---|---|---|
| Sign-in redirects fail or loop | Service worker intercepting `/__/auth/` | Check `navigateFallbackDenylist` includes `/^\/__\/auth/` |
| Service worker not updating | Old SW cached | Use `registerType: 'autoUpdate'` and set `Cache-Control: no-cache` on `sw.js` in firebase.json |
| App not installable on iOS | Safari doesn't support `beforeinstallprompt` | Add a manual "Add to Home Screen" instruction for iOS users |
| Firebase Storage images not loading offline | Not in runtime cache | Add the `firebasestorage.googleapis.com` runtime cache rule (already in Phase 1 config) |
| Push notifications not working | Wrong VAPID key or missing FCM SW | Verify VAPID key and check `firebase-messaging-sw.js` is in `public/` |
| "Could not find a valid build of workbox" error | Plugin installed in dependencies, not devDependencies | Run `npm install -D vite-plugin-pwa` |
| Icons not showing on iOS | Missing `apple-touch-icon` link tag | Add `<link rel="apple-touch-icon">` to `index.html` |
| Maskable icon looks wrong | Padding too small | Use 10–15% safe area padding around the logo |
